# Prepare shibuya-kafka-adapter 0.1.0.0 for initial release

This MasterPlan is a living document. The sections Progress, Surprises & Discoveries,

## Purpose / Big Picture

After this initiative, `shibuya-kafka-adapter 0.1.0.0` is ready to tag and upload to
Hackage. A downstream user can pull the library from Hackage, read a rendered README
and CHANGELOG, find a LICENSE in the tarball, and build against documented bounds.
The public surface is explicit (no module-wide re-exports), fatal Kafka errors are
surfaced rather than silently swallowed, GHC warnings match the standard used across
the sibling packages (`kafka-effectful`, `hw-kafka-streamly`), and a local contributor
can bring up Redpanda and create the topics exercised by the integration tests and
jitsurei examples with a single `just` recipe.

The audit that motivated this initiative is summarized in the Decision Log entry for
2026-04-18. Four coordinated ExecPlans address the findings. None of the plans has a
hard dependency on another; they can be implemented in any order, though two pairs
touch shared files and benefit from sequencing (see Integration Points).

## Surprises & Discoveries

- Audit on 2026-04-18 confirmed `Stream.mapMaybeM (… Left _err -> pure Nothing …)`
  at `Shibuya.Adapter.Kafka.Kafka.hs:84-89` masks fatal errors surviving
  `skipNonFatal`. The severity justifies inclusion as a blocker in Plan 5 rather
  than deferring to 0.2.

- Plan 4 (2026-04-18): local `cabal haddock` renders a per-module index, not a
  Hackage package landing page — so cabal metadata fields (`homepage`,
  `bug-reports`, `copyright`) are not visible locally. `cabal check` was used
  as a validation proxy since it applies Hackage's upload-time rules. The
  soft-dependency "Plan 7 → Plan 4" (run stricter warnings before polish) was
  not enforced in practice; Plan 4 was implemented first and caused no warning
  churn because the edits were purely metadata. Plan 7 remains free to proceed
  independently.


## Decision Log

- Decision: Decompose the 0.1.0.0 release-prep into four child ExecPlans rather than
  a single large plan.
  Rationale: The four concerns (metadata, correctness, API surface, developer
  ergonomics) are functionally distinct, independently verifiable, and can be worked
  on in any order. A single plan would mix publishing-cosmetics churn with a
  behavior change (fatal-error propagation), making the diff harder to review.
  Date: 2026-04-18

- Decision: Treat "no auto-resume after `AckHalt`" as a documentation change for
  0.1.0.0, not a runtime feature.
  Rationale: Implementing programmatic resume is a new adapter capability (an extra
  API on `Adapter` or a side channel from the handler). Scope creep for a point
  release. The current behavior (pause-until-session-end) is defensible if
  documented; a richer halt/resume story can land in 0.2.
  Date: 2026-04-18

- Decision: Keep `skipNonFatal` rather than `skipNonFatalExcept [isPartitionEOF]`
  for 0.1.0.0, and document the implication.
  Rationale: The adapter does not expose a bounded-read API, so EOF-driven
  termination isn't part of the public story. Documenting the caveat is
  sufficient; switching the filter is a behavior change better motivated by a
  concrete user need.
  Date: 2026-04-18

- Decision: Modeled after `hw-kafka-streamly`'s `Justfile` rather than inventing a
  new recipe layout.
  Rationale: Contributors moving between Kafka-related repos encounter the same
  commands. The topic names in `create-topics` come from the jitsurei examples and
  the tests' default topic pattern.
  Date: 2026-04-18
